# Apply42-Survey 安全威脅模型與修補路線圖

## 1. 文件資訊

- 專案：Apply42-Survey（Google Apps Script）
- 文件版本：v1.0
- 產出日期：2026-04-15
- 範圍：Web App（`doGet/doPost`）、`google.script.run` 伺服器函式、Spreadsheet 存取、快取、統計 API、日誌、郵件流程
- 目的：建立可執行的威脅模型與修補規劃，支援風險治理與驗收

## 3. 威脅模型表

評分說明：
- 緩解優先級 `P0`：立即修補（阻斷上線或緊急 hotfix）
- 緩解優先級 `P1`：高優先（1 週內）
- 緩解優先級 `P2`：中優先（1 個迭代內）
- 緩解優先級 `P3`：低優先（排程改善）

| ID | 威脅類型 | 攻擊前提 | 可能影響 | 偵測方式 | 緩解優先級 | 反制方式 |
|---|---|---|---|---|---|---|
| TM-01 | 權限繞過（IDOR / RPC 參數信任） | 攻擊者可在瀏覽器 console 呼叫 `google.script.run`，且伺服器函式信任前端傳入 teacher/user 物件 | 學生或低權限帳號可讀取他班資料（個資、志願） | 記錄呼叫者 email 與請求目標班級；偵測「呼叫者不屬於該班級」事件 | P0 | 伺服器端忽略前端身分物件；每次以 `Session + getUserData()` 重建身分；函式內強制 RBAC 檢查 |
| TM-02 | CSRF / 重放提交 | 使用者登入狀態存在，攻擊者可誘導送出 POST，或重放舊請求 | 非預期志願覆寫、資料污染、濫發郵件 | 偵測短時間同帳號重複提交、來源異常、nonce 重複 | P0 | 加入 CSRF token、一次性 nonce、`Origin/Referer` 驗證、重放窗口限制 |
| TM-03 | 部署存取範圍過寬 | Web App 設為 `ANYONE_ANONYMOUS` | 攻擊面大幅增加，被掃描與濫用機率上升 | 監控匿名請求比例、來源分佈、4xx/5xx 尖峰 | P0 | 存取改為 `DOMAIN` 或受控名單；僅允許學校網域帳號 |
| TM-04 | 列定位錯誤造成跨使用者寫入 | `findValueRow(email)` 命中非唯一列，或資料品質有重複 email | A 資料寫到 B 列（完整性破壞） | 寫入前檢查 email 命中列數；非唯一立即告警 | P0 | 建立 email 唯一性約束；命中列數必須等於 1；否則拒絕寫入 |
| TM-05 | 維運函式濫用（清快取 DoS） | 攻擊者可呼叫清快取函式 | 快取被清空導致延遲飆升、服務不穩 | 監控 cache clear 次數與來源帳號 | P1 | 維運函式加 admin ACL；操作審計；冷卻時間與速率限制 |
| TM-06 | 快取鍵碰撞（email 正規化） | 不同 email 正規化後同 key（如 `.` 與 `_`） | 跨帳號快取覆蓋、資料混淆/外洩 | 偵測 key 與原始 email 映射衝突 | P1 | 快取 key 改為 `SHA-256(email)`；快取值內含原 email 再次校驗 |
| TM-07 | 業務規則繞過（志願代碼僅做格式驗證） | 攻擊者可送 6 位數字但非合法選項 | 非法志願入庫、統計失真 | 監控不存在於選項表的代碼寫入 | P1 | 伺服器端比對 allowlist（依群類、學校上限） |
| TM-08 | 配額耗盡（郵件/Apps Script 配額） | 可自動化重複提交 | 郵件額度耗盡、服務降級 | 監控每帳號/每 IP 提交頻率與寄信量 | P1 | 速率限制、同內容去重、寄信節流、異常觸發暫停策略 |
| TM-09 | 錯誤詳情外洩（stack trace） | 攻擊者觸發例外 | 揭露內部結構，提升後續攻擊成功率 | 掃描回應是否含 stack 關鍵字 | P1 | 前台只回通用錯誤碼；詳細錯誤僅記內部安全日誌 |
| TM-10 | 過度日誌記錄敏感資料 | 系統記錄完整 request/user | 個資在日誌擴散 | 日誌 DLP 掃描姓名/學號/序號/信箱 | P1 | 日誌最小化、欄位遮罩、禁止記錄完整 payload |
| TM-11 | 統計資料枚舉與推測 | 可頻繁呼叫統計 API | 外部可推估群類填報趨勢 | 偵測連續遍歷 group/page 行為 | P2 | 統計 API 加角色控管、分級揭露、速率限制 |
| TM-12 | 競態條件（最後寫入覆蓋） | 同帳號多分頁/多裝置同時送出 | 資料一致性不穩定 | 偵測同帳號短秒級多寫入 | P2 | `LockService`、版本戳（optimistic lock）、提交序號 |
| TM-13 | 前端渲染 XSS 風險（配置/通知內容） | 管理端輸入含 HTML，前端用 `innerHTML` | 釣魚、使用者端腳本執行 | 前端 CSP report 與 DOM XSS 掃描 | P2 | 改 `textContent` 優先；必要 HTML 白名單 sanitizer；嚴格 CSP |
| TM-14 | 權限模型漂移（老師跨班查看） | 老師資料欄配置錯誤或逗號分班維護錯誤 | 越權查看其他班級 | 稽核「老師-班級映射」異常變更 | P2 | 權限資料正規化；管理後台加檢核與雙人覆核 |

## 4. 攻擊情境（Scenario）摘要

### Scenario A：學生越權讀取班級資料
1. 學生登入後開啟瀏覽器 console。  
2. 直接呼叫可存取資料的 `google.script.run` 函式，手動傳入其他班級參數。  
3. 伺服器若信任該參數，回傳敏感資料。  

防線：伺服器端身分重建 + RBAC 必做，不接受前端傳入權限上下文。

### Scenario B：重放提交造成配額耗盡
1. 取得一次合法提交請求格式。  
2. 批次重送，觸發大量寫入與寄信。  
3. 導致信件配額耗盡、正常流量受影響。  

防線：CSRF + nonce + 速率限制 + 郵件去重與節流。

### Scenario C：資料列不唯一造成跨使用者覆寫
1. 試算表存在重複 email。  
2. 系統只抓第一筆命中列。  
3. 寫入錯列，造成完整性與可追溯性問題。  

防線：唯一性檢查與寫入前一致性斷言，違規即拒絕。

## 5. 修補路線圖（里程碑）

## M1：身分與授權防線（P0，目標 3 天）

### 工作項目
- Web App 存取範圍調整為受控（至少網域內）。
- 所有伺服器函式一律由伺服器端取得當前登入者，不接受前端身分物件。
- 建立統一 RBAC guard（學生、老師、管理者）。
- 對寫入流程加「唯一列命中」斷言。

### 驗收條件
- 學生帳號無法讀取任意班級資料。
- 匿名或非授權帳號無法進入資料 API。
- 寫入時若 email 命中列數不為 1，請求被拒絕並有稽核紀錄。

### 回歸測試清單
- 學生正常提交與查詢流程可用。
- 老師僅能看授權班級。
- 管理功能不受影響。
- 已有資料與統計頁載入正常。

## M2：請求完整性與抗重放（P0/P1，目標 3 天）

### 工作項目
- 新增 CSRF token 與一次性 nonce 驗證。
- 驗證 `Origin/Referer`。
- 對提交與寄信加速率限制與短期去重。

### 驗收條件
- 無 token 或錯誤 token 一律拒絕。
- 相同 nonce 第二次提交必拒絕。
- 壓測下不會出現異常寄信暴增。

### 回歸測試清單
- 正常使用者單次提交成功。
- 重新整理頁面後 token 正常更新。
- 倒數截止前後流程符合預期。

## M3：輸入與業務規則強化（P1，目標 4 天）

### 工作項目
- 志願代碼做伺服器端 allowlist 驗證（依群類與可報名規則）。
- 針對異常參數回傳標準化錯誤碼。
- 前端動態內容輸出改 `textContent` 優先，必要 HTML 走白名單。

### 驗收條件
- 非法但格式正確的代碼被拒絕。
- 前端無 DOM XSS 發現。
- 錯誤回應不暴露內部細節。

### 回歸測試清單
- 合法志願提交成功。
- 非法志願顯示可理解錯誤訊息。
- 統計頁與老師頁資料渲染正確。

## M4：可觀測性與稽核（P1/P2，目標 4 天）

### 工作項目
- 安全稽核日誌：記錄 actor、action、resource、result、reason code。
- 移除或遮罩敏感日誌欄位。
- 建立安全告警規則（越權、重放、速率異常、cache clear 異常）。

### 驗收條件
- 可追蹤每次敏感操作來源與結果。
- 日誌掃描不再出現明碼個資 payload。
- 告警可在 5 分鐘內通知維運。

### 回歸測試清單
- 正常流量不誤報。
- 模擬異常操作可正確告警。
- 日誌查詢可回溯事件鏈。

## M5：一致性與韌性（P2，目標 3 天）

### 工作項目
- 寫入鎖（`LockService`）與版本戳，避免競態覆蓋。
- 快取 key 改雜湊，補 key collision 監控。
- 維運函式增加雙重確認與權限分層。

### 驗收條件
- 同帳號並發提交不會出現不一致。
- 無快取鍵碰撞資料汙染。
- 維運函式僅授權人員可執行。

### 回歸測試清單
- 並發提交測試通過。
- 快取命中與失效策略符合預期。
- 系統效能維持在可接受範圍。

## 6. 安全測試矩陣（建議）

| 類別 | 測試重點 | 通過標準 |
|---|---|---|
| 認證/授權 | 學生越權讀取、老師跨班讀取、匿名存取 | 全數被拒絕，回傳一致錯誤碼 |
| 請求完整性 | CSRF、重放、nonce 重用 | 偽造與重放請求不可成功 |
| 輸入驗證 | 非法志願代碼、超長參數、惡意字串 | 不寫入、不崩潰、有清楚錯誤 |
| 資料完整性 | 重複 email、並發寫入 | 不會錯列覆寫，衝突有告警 |
| 資訊外洩 | stack trace、敏感日誌 | 前台無堆疊，日誌已遮罩 |
| 可用性 | 快取清除濫用、提交洪水 | 服務可維持，攻擊被節流 |

## 7. 維運監控與告警規則（最小集）

- 規則 R1：同帳號 5 分鐘內提交次數 > 閾值，觸發 `High` 告警。
- 規則 R2：任一帳號出現 `RBAC_DENY` 連續多次，觸發 `Medium` 告警。
- 規則 R3：cache clear 操作在 10 分鐘內 > 閾值，觸發 `High` 告警。
- 規則 R4：寄信量短時間異常突增，觸發 `High` 告警。
- 規則 R5：回應中出現 stack 關鍵字（灰箱檢測），觸發 `High` 告警。

---

## 8. 風險接受與例外管理

- 任何 `P0` 不接受延期上線。
- `P1` 若需延後，必須有：
  - 風險接受人（具責任職務）
  - 到期日
  - 補償控制（temporary mitigation）
  - 追蹤工單

---

## 9. 建議下一步

1. 先執行 `M1 + M2`，建立最基本防線。  
2. 同步建立安全稽核日誌格式，避免修補後無法驗證效果。  
3. `M3~M5` 依迭代排程完成，並在每個里程碑做一次滲透測試回歸。
