## Context

目前系統雖有一般 Logger，但缺乏安全事件語意與偵測門檻，且含過度紀錄敏感資訊風險。需要在不破壞既有功能下建立最小可行的安全監控框架。

## Goals / Non-Goals

**Goals:**
- 建立可追蹤、可告警的安全事件模型。
- 建立敏感端點節流策略，抑制濫用。
- 使事件可支援事後鑑識與即時處置。

**Non-Goals:**
- 不導入大型 SIEM 平台重構。
- 不追求一次到位的全域監控，先覆蓋高風險路徑。

## Decisions

- Decision 1: 安全事件採統一欄位格式（actor/action/resource/result/reason）。
- Decision 2: 敏感資料寫入日誌前一律遮罩或移除。
- Decision 3: 對提交、清快取、授權拒絕等事件設定告警規則。

## Risks / Trade-offs

- [節流門檻過嚴誤傷合法流量] → 先以觀察值上線，逐步收斂。
- [告警噪音過多] → 設置分級與聚合策略。
- [事件紀錄不足] → 先確保最小核心欄位完整，再逐步擴充。
