## ADDED Requirements

### Requirement: 階段化 UI 一致化執行
系統 SHALL 在同一個變更中，依序執行 A→B→C 三個階段，不得跳階實作。

#### Scenario: 實作順序約束
- **WHEN** 團隊啟動此變更實作
- **THEN** 必須先完成階段 A 驗收，才可進入階段 B
- **AND** 必須先完成階段 B 驗收，才可進入階段 C

### Requirement: 階段 A 快速一致化
系統 SHALL 先統一三個入口頁面的基準視覺與狀態文案，以降低使用者理解成本。

#### Scenario: 基準元件一致
- **WHEN** 使用者在學生頁、老師頁、統計對話框切換
- **THEN** 卡片、按鈕、表單、alert、表格的視覺層級與操作回饋應一致

#### Scenario: 狀態文案一致
- **WHEN** 系統顯示載入中、空資料、錯誤、已截止等狀態
- **THEN** 三個入口應使用一致語氣與明確行動指引

### Requirement: 志願驗證改為非阻斷回饋
系統 MUST 以非阻斷式驗證回饋取代阻斷式 alert，並能在手機上正常運作。

#### Scenario: 重複志願回饋
- **WHEN** 學生選到兩個相同志願
- **THEN** 系統應在對應欄位顯示錯誤提示
- **AND** 頁首顯示錯誤摘要
- **AND** 送出按鈕進入停用狀態並顯示修正提示

#### Scenario: 超過校系上限回饋
- **WHEN** 學生選填志願超過大學規定上限
- **THEN** 系統應顯示可定位的欄位錯誤與摘要提示
- **AND** 不應使用阻斷式 alert 中斷操作

#### Scenario: 手機可用性
- **WHEN** 使用者以手機操作志願欄位
- **THEN** 錯誤訊息與修正操作在可視範圍內
- **AND** 主要可點擊區域符合最小觸控尺寸

### Requirement: 階段 B 設計 token 骨架
系統 MUST 建立共用 design tokens 並讓三個入口頁面引用，避免樣式漂移。

#### Scenario: token 共用生效
- **WHEN** 團隊調整主色或字級 token
- **THEN** 三個入口頁面應同步反映變更
- **AND** 不需在三個檔案分別手動調整重複定義

### Requirement: 階段 C 統計體驗一致
系統 SHALL 對齊三個統計入口的互動節奏與狀態處理。

#### Scenario: 統計操作節奏一致
- **WHEN** 使用者在任一入口執行統計查看、篩選與展開更多
- **THEN** 操作流程與回饋節奏應保持一致，不需重新學習

#### Scenario: 驗收矩陣可追蹤
- **WHEN** 團隊完成階段 C
- **THEN** 應提供三入口一致性驗收矩陣，記錄每個場景的驗證結果
