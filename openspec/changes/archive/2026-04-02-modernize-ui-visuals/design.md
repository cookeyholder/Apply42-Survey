## Context

目前三個主要前端入口（學生頁、老師頁、統計對話框）雖然配色接近，但樣式來源分散、互動節奏不一致。此變更目標是同一提案內完成三條改善建議，並明確採用 A→B→C 的順序降低風險。

## Goals / Non-Goals

**Goals:**
- 階段 A：先收斂視覺一致性與可讀性，快速提升操作體驗。
- 階段 B：建立可維護的設計 token 骨架，降低後續改動成本。
- 階段 C：統一統計三入口的使用節奏與狀態顯示，降低認知負擔。
- 將志願驗證從阻斷式 alert 改為非阻斷、可定位、可恢復的回饋模式，優先保證手機可操作性。

**Non-Goals:**
- 不在本 change 引入新前端框架。
- 不重做後端 API 或資料模型。
- 不一次性大改全部頁面版型。

## Decisions

- 決策 1：採用分階段實作（A→B→C），禁止跳階。
   - 理由：先確保低風險可見成果，再處理架構化與流程整合。
   - 替代方案：一次完成所有改動；回歸風險高且難定位問題。

- 決策 2：以 design token 驅動樣式，不再複製貼上三份 `:root`。
   - 理由：減少 drift 與後續維護成本。
   - 替代方案：維持各頁獨立樣式；短期快、長期成本高。

- 決策 3：統計流程統一以「同一組狀態模型」描述。
   - 理由：使用者在三入口切換時不需重學操作方式。
   - 替代方案：保持現況；會持續出現 UX 斷裂。

- 決策 4：志願驗證採用三層回饋模型（欄位內提示 + 頁首摘要 + 送出按鈕狀態）取代 alert。
   - 理由：alert 會中斷輸入流程且在手機上易造成操作跳躍，三層回饋可直接指出錯誤位置與修復方向。
   - 替代方案：只用 toast；提示容易被忽略且無法精準對應欄位。

- 決策 5：手機優先互動規範納入驗收門檻（觸控區、字級、可視區、即時清錯）。
   - 理由：本系統高比例在手機情境操作，驗證回饋若不手機優先會直接降低完成率。
   - 替代方案：沿用桌機優先；會造成手機誤觸與回饋不清。

## Risks / Trade-offs

- [階段 B 提前導入造成變更面過大] → 嚴格先完成階段 A 驗收再進 B。
- [三入口流程統一引發行為回歸] → 階段 C 需建立完整手動驗收矩陣。
- [token 共用後樣式覆寫衝突] → 定義 token 命名與覆寫層級，避免隱式競爭。
- [驗證提示過多造成視覺噪音] → 採「欄位錯誤優先、摘要次之、toast 選配」策略並限制同時顯示數量。
- [手機上錯誤提示被鍵盤遮住] → 驗證時自動聚焦首個錯誤欄位，並確保錯誤訊息在可視範圍內。

## Migration Plan

1. 階段 A：統一基準 UI 元件樣式與狀態文案（低風險）。
   - A-1：以非阻斷驗證回饋取代 alert（重複志願、超上限）。
   - A-2：完成手機驗收（最小點擊區、錯誤字級、按鈕狀態、可視區）。
2. 階段 B：抽取並套用共用 token/樣式片段（中風險）。
3. 階段 C：對齊三入口統計流程與狀態（高價值）。
4. 每階段完成後皆執行回歸清單與可回退方案驗證。
