## Context

目前 `index.html` 與 `teacherView.html` 各自內嵌大段統計 JavaScript，邏輯高度重複，導致修補審查意見時常出現雙邊不一致。先前嘗試抽共用檔時，曾發生腳本注入位置錯誤（混入 `<style>`）與舊監聽器殘留（`loadStatsSummary` 作用域失效）等回歸，顯示此變更具生命週期與模板整合複雜度。

## Goals / Non-Goals

**Goals:**
- 建立單一統計前端共用模組，讓學生端與老師端共用同一套狀態與渲染流程。
- 以明確初始化介面管理事件綁定，避免全域函式依賴與重複監聽。
- 確保腳本注入位置可驗證（僅允許注入 `<script>` 區塊），避免模板解析錯誤。
- 保持現有後端 API 與使用者可見行為一致（summary-first、group-detail、Top N、展開更多）。

**Non-Goals:**
- 不改動統計後端資料模型與 API 介面。
- 不在本次變更中重做統計視覺設計與欄位結構。
- 不處理與統計無關的前端重構。

## Decisions

- 決策 1：採用 `statisticsShared.html` 作為共用腳本來源。
  - 理由：符合 GAS `HtmlService.createHtmlOutputFromFile(...).getContent()` 既有模式，可維持原模板系統。
  - 替代方案：改成 `.gs` 產生字串或改單頁架構；成本高且偏離現有專案慣例，暫不採用。

- 決策 2：共用模組對外只暴露單一初始化呼叫（例如 `initProgressiveStatisticsTab()`），不暴露內部載入函式。
  - 理由：可避免 `loadStatsSummary` 這類內部函式被外部舊監聽器直接呼叫而失效。
  - 替代方案：掛載多個全域 API；易造成耦合與未授權呼叫，暫不採用。

- 決策 3：以「一次綁定、可重入保護」處理 tab 事件。
  - 理由：防止多次切換頁籤重複註冊監聽，避免重複請求與效能退化。
  - 替代方案：每次切頁重建監聽；邏輯複雜且易重複綁定，暫不採用。

- 決策 4：保留頁面端差異（學生/老師資料面板）在原檔，僅抽統計共用區。
  - 理由：可維持最小改動範圍，降低回歸風險。
  - 替代方案：整頁共用；風險過高，不符合本次目標。

## Risks / Trade-offs

- [腳本注入位置錯誤] → 加入檢查清單：共用檔只能在頁尾 `<script>` 區塊載入，禁止出現在 `<style>`。
- [舊事件監聽殘留] → 移除舊 `shown.bs.tab` 監聽，統一由共用初始化管理。
- [初始化時機不一致] → 初始化時檢查必要 DOM 是否存在，缺件時顯示可診斷訊息並安全終止。
- [雙頁行為偏移] → 以同一組驗收場景（首次載入、切換群類、展開更多、錯誤訊息）在兩頁都驗證。

## Migration Plan

1. 新增 `statisticsShared.html`，先複製既有統計邏輯並包裝為單一初始化介面。
2. 在 `index.html` 與 `teacherView.html` 移除重複統計腳本，改為載入共用檔並呼叫初始化。
3. 移除舊有統計 tab 監聽，避免與共用模組重複綁定。
4. 進行手動回歸驗證（學生端、老師端）。
5. 若異常，回退為雙檔內嵌版本（保留回退 commit）。
