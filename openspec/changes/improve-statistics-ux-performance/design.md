## Context

現況統計頁在老師端與學生端均採延遲載入，但統計資料仍一次回傳全量群類與志願統計，再由前端一次性渲染。當資料列數上升時，主要延遲來自三個階段：工作表掃描、payload 傳輸、DOM 大量建立。現有快取可降低重複讀取成本，但無法改善首次全量載入的等待體感。

## Goals / Non-Goals

**Goals:**
- 將統計頁首屏改為摘要先行，降低首次等待時間。
- 將群類明細改為按需載入，避免不必要的全量資料傳輸。
- 建立志願代碼到校系名稱的一致映射，統一顯示語意。
- 定義可驗收的效能預算，避免後續功能擴充導致回歸。

**Non-Goals:**
- 本變更不引入新的外部圖表函式庫。
- 本變更不調整統測原始報名資料的欄位結構。
- 本變更不改動統計頁以外的主要業務流程。

## Decisions

1. 採用雙層資料端點策略（摘要端點 + 群類明細端點）。
- 理由：摘要資料體積小、可快速返回，明細僅在需要時載入。
- 替代方案：維持單一全量端點。捨棄原因：資料量增長時等待時間不可控。

2. 統計顯示資料以語意化欄位輸出（choiceCode, choiceName, choiceLabel）。
- 理由：前端不需重複查表，可減少渲染前處理並降低顯示不一致風險。
- 替代方案：前端即時映射。捨棄原因：老師端與學生端會重複運算，且快取命中不穩。

3. 視覺化優先策略為 Top N 長條圖 + 可排序明細表。
- 理由：可讀性高、渲染成本低、對帳容易。
- 替代方案：預設多圖並行渲染（環狀圖/熱力圖/堆疊圖）。捨棄原因：首屏成本與認知負擔偏高。

4. 快取分層：摘要短 TTL、映射中長 TTL、群類明細中 TTL。
- 理由：摘要更新頻率最高且需快速可見；映射資料相對穩定可延長快取。
- 替代方案：單一 TTL。捨棄原因：無法同時兼顧一致性與效能。

## Risks / Trade-offs

- [摘要與明細在快取更新時出現短暫不一致] → 以資料版本戳與生成時間標記，前端提示最後更新時間。
- [群類過多時仍可能造成切換卡頓] → 限制預設 Top N 並採分頁/分批渲染。
- [校系映射來源欄位異動導致顯示錯誤] → 啟動時檢查必要欄位，缺欄時回傳可觀測錯誤訊息。
- [老師端與學生端雙模板維護成本增加] → 抽共用資料格式與渲染約定，避免分支邏輯漂移。

## Migration Plan

1. 先新增摘要與群類明細資料結構，不移除既有統計輸出。
2. 前端切換為先取摘要、再取指定群類明細；保留舊清單渲染 fallback。
3. 上線後觀測首屏時間與錯誤率，再逐步關閉舊全量路徑。
4. 若出現異常，快速回切到舊全量統計路徑。

## Open Questions

已確認決策：
- 摘要卡顯示「近 X 分鐘更新」，且以資料生成時間為計算基準。
- 群類明細預設 Top N = 10，Top 10 以外提供「展開更多」或「翻頁」機制。
- 摘要與明細回應加入 version 欄位供前端判斷資料一致性。
- 不新增老師端「匯出目前篩選明細」。
