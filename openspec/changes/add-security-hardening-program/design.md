## Context

專案存在多個高風險安全議題（授權繞過、重放、資料一致性、可觀測性不足）。這些議題分佈於不同模組，且彼此有依賴關係。若不先建立治理框架，實作順序與驗收標準容易失焦。

## Goals / Non-Goals

**Goals:**
- 建立可追蹤的父子提案治理模型。
- 明確定義分工、依賴、里程碑與驗收關卡。
- 降低單次變更風險，提升審查與回歸效率。

**Non-Goals:**
- 不在父提案直接修改應用程式業務邏輯。
- 不以父提案取代子提案的技術設計細節。

## Decisions

- Decision 1: 採「1 父提案 + 4 子提案」架構。
  - Rationale: 將 P0/P1 風險拆為可獨立交付單位，避免大型提案阻塞。
  - Alternatives considered:
    - 單一大提案：文件與驗收負擔過大，回歸風險高。
    - 完全獨立提案無父層：缺乏全局門檻，難管理優先級。

- Decision 2: 子提案以「可獨立驗收」為原則。
  - Rationale: 每個提案皆可完成閉環（需求、設計、任務、測試）。

- Decision 3: 設定階段性關卡。
  - Rationale: 先落地 P0（授權與請求完整性），再進行 P1/P2（一致性、觀測、防護）。

## Risks / Trade-offs

- [跨提案依賴未明確] → 以父提案 tasks 明列執行順序與 gate。
- [子提案交付時序落差] → 每個里程碑加上最小可驗收標準（MVA）。
- [文件與實作漂移] → 每次子提案合併前需更新父提案追蹤狀態。

## Migration Plan

1. 先完成父提案 artifacts，確認治理範圍。
2. 啟動 4 個子提案，依序執行：RBAC → CSRF/重放 → 一致性/快取 → 觀測/節流。
3. 每完成一個子提案即更新父提案追蹤清單。
4. 全部子提案驗收通過後，進入整體安全回歸與收斂。

## Open Questions

- 是否需要再拆出「前端輸出安全（XSS/CSP）」為第 5 個子提案？
- 是否需在父提案新增跨提案 KPI（例如誤報率、告警平均處置時間）？
