## Context

目前專案為 Google Apps Script 單體應用，前後端邏輯分散於多個 `.gs` 與 `.html` 檔案。先前審查發現的問題同時涉及統計計算、前端輸出安全、回傳穩定性與資料讀取效能，且問題分布於多個模組，屬於跨檔案且具安全與效能風險的修復。

本次變更不新增外部相依套件，不調整對外 API，目標是在最小行為擾動下完成可驗證的集中修補，並以高頻提交降低回歸風險。

欄位盤點來源已固定於 `worksheet-columns.md`，實作前先以該文件核對「考生志願列表」與「參數設定」實際欄位。

## Goals / Non-Goals

**Goals:**
- 以固定優先序修復審查問題，先解決正確性與安全性，再處理穩定性與效能。
- 在前端輸出路徑建立一致的安全策略，避免設定值與錯誤訊息直接注入 HTML。
- 讓 `doGet` / `doPost` 的回傳與變數作用域可預期，避免隱含全域狀態。
- 降低 `getOptionData` 的全表掃描成本，並恢復統計查詢快取以減少重算。
- 導入高頻提交策略，確保每批修補可獨立驗證與回滾。

**Non-Goals:**
- 不重構整體 UI 架構或模板引擎。
- 不改變 Google Sheets 欄位結構與欄位命名。
- 不引入新的權限模型或登入流程。

## Decisions

### Decision 1: 先修正統計索引邏輯，再進行其他修補
- 決策：先修正 `statistics.gs` 內代碼/名稱欄位讀取索引一致性，確保統計結果正確。
- 理由：統計錯誤屬於資料正確性問題，風險最高且會影響使用者決策。
- 替代方案：先做前端安全修補。
- 不採用原因：即使前端更安全，錯誤統計仍會持續輸出錯誤結果。

### Decision 2: 前端動態文字改採安全輸出策略
- 決策：對使用者可見動態文字優先使用 `textContent`，僅在必要時使用模板字串並搭配 escape。
- 理由：可明確降低 XSS 風險，且不需引入額外函式庫。
- 替代方案：保留 `innerHTML` 並加強 regex sanitize。
- 不採用原因：regex sanitize 容易遺漏邊界條件，長期維護風險高。

### Decision 3: 修補穩定性缺陷（作用域與回傳路徑）
- 決策：將隱含全域變數改為區域宣告，補齊未知 userType 的明確回應。
- 理由：可避免跨呼叫污染與不可預期空回應。
- 替代方案：僅加 Logger，不改邏輯。
- 不採用原因：只記錄不修正會保留潛在故障點。

### Decision 4: 局部化資料讀取並恢復統計快取
- 決策：避免每次全表掃描；優先使用索引/目標列讀取，統計資料恢復短時快取並配合手動清除快取機制。
- 理由：在資料量成長時可明顯改善延遲與執行配額壓力。
- 替代方案：維持現狀僅加註解。
- 不採用原因：無法解決實際效能瓶頸。

### Decision 5: 採高頻提交策略（每個風險主題一提交）
- 決策：以「統計正確性 → 前端安全 → 穩定性 → 效能」切分提交，每個提交附最小驗證。
- 理由：降低單次變更面積，便於 code review、定位問題與快速回滾。
- 替代方案：單一大提交。
- 不採用原因：回歸定位與風險控管成本過高。

## Risks / Trade-offs

- [前端輸出策略變更可能影響既有 HTML 呈現] → 以視覺回歸檢查重要區塊（標題、頁尾、錯誤訊息）。
- [資料讀取路徑調整可能與工作表欄位差異衝突] → 以欄位存在檢查與安全 fallback 保底。
- [快取恢復可能造成短暫資料延遲] → 設定短 TTL 並保留「清除快取」管理入口。
- [高頻提交增加操作次數] → 每次提交僅處理單一主題，降低審查與測試複雜度。

## Migration Plan

1. 先在 `statistics.gs` 修復索引邏輯並驗證統計結果。
2. 於 `index.html`、`teacherView.html` 修正高風險 `innerHTML` 輸出與錯誤訊息渲染。
3. 於 `main.gs` 修復變數宣告與未知 userType 回傳路徑。
4. 於 `retrieveData.gs`、`statistics.gs` 完成資料讀取優化與統計快取恢復。
5. 依主題高頻提交，每次提交後執行對應驗證並記錄結果。
6. 如任一提交引發異常，以該提交為單位回滾，不影響其他已驗證主題。

## Acceptance Criteria

- 統計邏輯驗收：代碼模式與名稱回退模式下，統計結果均與人工抽查一致。
- 安全輸出驗收：學生端與老師端高風險動態文字不直接注入不可信 HTML。
- 錯誤顯示驗收：統計錯誤區塊可正確渲染，錯誤文字可見且不觸發腳本執行。
- 穩定性驗收：`doGet` 對未知角色提供明確回應，`doPost` 不產生隱含全域狀態。
- 效能驗收：資料讀取避免每次全表掃描，統計快取可命中且可由管理入口清除。
- 交付驗收：至少四個高頻提交，且每個提交可獨立驗證與回滾。

## Open Questions
