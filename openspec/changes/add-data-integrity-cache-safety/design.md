## Context

系統以試算表作為資料來源，若索引條件不嚴謹會導致誤寫；快取鍵若正規化過度則會碰撞。需強化資料與身份映射的一致性保證。

## Goals / Non-Goals

**Goals:**
- 保證提交只會寫入唯一正確目標列。
- 防止非法代碼進入資料庫。
- 降低快取鍵碰撞造成的跨帳號風險。

**Non-Goals:**
- 不重寫整個快取框架。
- 不變更統計計算邏輯本身。

## Decisions

- Decision 1: 寫入前執行唯一命中斷言（0 或 >1 皆拒絕）。
- Decision 2: 伺服器端代碼驗證使用動態 allowlist。
- Decision 3: 快取 key 使用 email 雜湊（含原值校驗欄位）。

## Risks / Trade-offs

- [資料品質問題導致拒絕率上升] → 補資料清理流程與管理告警。
- [allowlist 建置成本] → 先以現有資料來源生成，後續優化快取。
- [快取遷移期命中率下降] → 設計過渡策略與雙讀短期機制。
